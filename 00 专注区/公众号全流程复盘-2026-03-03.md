## 步骤 2：配图 ❌

### 使用的 Skill

`imagen`（基于 Google Gemini API 的图片生成技能）

### 失败原因（两轮尝试）

**第一轮：缺少 Gemini API Key**
- imagen Skill 使用 Google Gemini API（`generativelanguage.googleapis.com`），需要环境变量 `GEMINI_API_KEY`
- Gemini API Key 格式通常以 `AIza` 开头，从 https://aistudio.google.com/ 获取（免费）
- 最初提供的 key（`sk-or-v1-...`）是 **OpenRouter 的 API Key**，不是 Gemini Key，无法用于 Gemini API

**第二轮：Gemini API 配额耗尽（429 RESOURCE_EXHAUSTED）**
- 提供了正确的 Gemini Key（`AIza` 开头）后，API 返回 429 错误
- 具体错误：`"You exceeded your current quota"` / `"status": "RESOURCE_EXHAUSTED"`
- Gemini 免费版有每日请求上限，该 key 当天的配额已经用完
- 并行发送 3 张图的请求 + 后续串行重试，全部被拒绝
- 配额监控页面：https://ai.dev/rate-limit

### 解决方案

**方案 A：等配额重置后重新生成（最简单）**
1. Gemini 免费版配额通常每天自动重置
2. 设置环境变量：`export GEMINI_API_KEY='AIzaSyBWaKQ40OWzWYxHXaR71B-qdzo11VMd4HE'`
3. 逐张生成（不要并行，避免再次打满配额）：
   ```bash
   python ".claude/skills/imagen/scripts/generate_image.py" "提示词" "输出路径.png"
   ```

**方案 B：在公众号后台用 AI 配图**
- 配图提示词已写入文章 Markdown 的 HTML 注释中（3 处）
- 在公众号编辑器中，找到对应位置，使用后台的 AI 配图功能，粘贴提示词生成

**方案 C：升级 Gemini API 计划**
- 免费版配额有限，如需频繁生成图片，考虑付费计划
- 查看计划详情：https://ai.google.dev/gemini-api/docs/rate-limits

## 步骤 3：排版 ✅

### 使用的 Skill

`06 我的工具/jimliu-baoyu-markdown-to-html/`

### 做了什么

1. 选择 `default` 主题（标题居中带底边，二级标题白字彩底）
2. 运行转换命令：
   ```bash
   npx -y bun "06 我的工具/jimliu-baoyu-markdown-to-html/scripts/main.ts" "文章.md" --theme default
   ```
3. 输出 HTML 文件到同目录

### 踩坑：首次运行需安装依赖

首次运行报错 `Cannot find module 'unist-util-visit-parents/do-not-use-color'`。

**解法：** 在 `scripts/md/` 目录下执行 `npx -y bun install` 安装依赖包。安装后运行正常。

```bash
cd "06 我的工具/jimliu-baoyu-markdown-to-html/scripts/md" && npx -y bun install
```

### 注意事项

- 3 个主题可选：`default`（正式）、`grace`（高端）、`simple`（极简）
- 文章中的 HTML 注释（配图提示词）不会出现在最终 HTML 中
- 如果 Markdown 中有中文加粗（`**中文**`），排版工具会自动处理 CJK 兼容性

## 步骤 4：发布 ✅（经排查后成功）

### 使用的 Skill

`06 我的工具/jimliu-baoyu-post-to-wechat/`

### 排查过程（共 5 轮尝试）

| 轮次 | 错误 | 原因 | 解法 |
|------|------|------|------|
| 1-2 | 502 Bad Gateway | 系统 HTTP 代理（127.0.0.1:7890）拦截了对 Chrome 调试端口的本地请求 | 加 `no_proxy=127.0.0.1,localhost` |
| 3 | Unable to connect | 之前尝试遗留的 Chrome 进程 + SingletonLock 文件 | 杀进程 + 删锁文件 |
| 4 | Element not found: .ProseMirror | 编辑器加载时序问题（实际选择器是对的） | 添加多选择器备选 + 重试逻辑 |
| 5 | ✅ 成功 | — | 完整流程跑通，草稿已保存 |

### 真正的根因：系统代理

**核心问题**：macOS 系统设置了 HTTP 代理 `127.0.0.1:7890`（Web 代理 + 安全 Web 代理均开启），同时环境变量 `http_proxy` 和 `https_proxy` 也指向该地址。

发布脚本通过 `fetch('http://127.0.0.1:{port}/json/version')` 连接 Chrome 调试端口时，请求被代理拦截 → 代理不认识这个本地请求 → 返回 502 Bad Gateway。

**解法**：运行发布命令时加 `no_proxy` 环境变量：

```bash
no_proxy="127.0.0.1,localhost" NO_PROXY="127.0.0.1,localhost" npx -y bun ...
```

### 脚本修改记录

修改了 `wechat-article.ts` 两处，增强编辑器选择器的兼容性：

1. **第 296-310 行**：点击编辑器区域时，尝试多个 CSS 选择器（`.ProseMirror`、`[contenteditable="true"]`、`.edui-editor-body`、`#ueditor_0`），最多重试 3 次
2. **第 180 行**：`selectAndReplacePlaceholder` 函数中，编辑器查询也使用多选择器备选


### 注意事项

- 首次运行需扫码登录微信公众号，session 会保存到 `~/.local/share/wechat-browser-profile/`
- **必须关闭所有 Chrome 窗口**后再运行（macOS 上 Chrome 不支持同时运行多实例）
- **必须加 `no_proxy`**，否则代理会拦截本地请求
- 脚本会保存为草稿，不会自动发布，需要手动确认发布

## 后续改进方向

### 最高优先级：效率问题

本次全流程在 AI 对话中串行执行，耗时约 30 分钟，token 消耗约 20 元。**排版和发布环节完全不需要 AI 参与**，却在对话中来回消耗 token，这是最大的浪费。

**改进方案：写一个 `publish.sh` 一键脚本**

将排版 + 发布自动化为一行命令，脱离 AI 对话：

```
输入：markdown 文件路径
自动：排版（转 HTML）→ 发布（保存草稿到公众号后台）
```

改进后的工作流：

| 环节 | 谁做 | 怎么做 | 花费 |
|------|------|--------|------|
| 写作 | AI | 聊素材 → 写文章 → 确认 → AI 环节结束 | token（值得花） |
| 配图 | 自己 | 公众号后台 AI 配图（免费、快、效果可控） | 0 |
| 排版 + 发布 | 一键脚本 | `./publish.sh 文章.md`，30 秒出草稿 | 0 |

预估改进效果：
- 时间：30 分钟 → 写作 5-10 分钟 + 脚本 30 秒
- 成本：~20 元 → 写作部分的 token 费用（预计 3-5 元）

### 其他改进

1. **配图方案调整**：放弃 Gemini API（配额限制 + 额外成本），直接用公众号后台自带的 AI 配图功能
2. **中文路径兼容**：发布 Skill 的 `import.meta.url` 中文编码问题需修复（`parseMarkdownWithPlaceholders` 函数中加 `decodeURIComponent`）
3. **编辑器选择器兼容**：已修复（本次新增多选择器备选），后续关注微信公众号编辑器更新
