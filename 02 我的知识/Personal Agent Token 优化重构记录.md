# Personal Agent Token 优化重构记录

## 📋 背景

### 问题发现
2026-03-05，用户反馈 token 消耗过高，希望优化系统的 token 使用效率。

### 初始状态
- CLAUDE.md：~435 行，每次会话自动加载
- 启动序列：每次加载大量文件（SOUL.md, USER.md, MEMORY_LOG 全部等）
- 无分层加载机制
- 知识库与项目工作混在一起

---

## 🎯 优化目标

1. **降低日常 token 消耗**：减少 50-70%
2. **保持功能完整性**：深度场景不妥协
3. **提升响应速度**：轻量模式快速启动
4. **架构清晰化**：知识库与项目分离

---

## 🔧 实施的优化措施

### 第一阶段：启动序列优化

#### 1. SOUL.md 缓存机制
**问题**：每次会话都重新加载 SOUL.md（~2000 tokens）

**方案**：
```
- 首次会话：加载 SOUL.md
- 后续会话：检查文件修改时间
- 未变更：跳过加载
- 有变更：重新加载
```

**实施**：
- 修改 CLAUDE.md 第 53 行
- 在 MEMORY_LOG.md 记录上次加载时间戳

**效果**：非首次会话节省 ~2000 tokens

---

#### 2. MEMORY_LOG.md 限量读取
**问题**：读取"尾部"可能很长，随着使用时间增长越来越大

**方案**：
```
- 只读取最近 3 条记录
- 使用 tail -n 50 确保完整性
- 解析出独立条目
```

**实施**：
- 修改 CLAUDE.md 第 52 行
- 从"读取尾部"改为"读取最近 3 条"

**效果**：节省 ~500-1000 tokens（取决于历史长度）

---

#### 3. 人生红点提醒降频
**问题**：每次启动都读取 USER.md 并触发人生红点提醒

**方案**：
```
- 改为仅周一执行
- 非周一跳过步骤 7
- 避免每次加载 USER.md
```

**实施**：
- 修改 CLAUDE.md 第 55 行和第 87-92 行
- 增加周一判断逻辑

**效果**：周二至周日节省 ~2000 tokens

---

### 第二阶段：增量同步机制

#### 4. Git Diff 增量检测
**问题**：git pull 后全量扫描所有文件

**方案**：
```bash
# 步骤 1 增强
git pull
git diff --name-only HEAD@{1} HEAD  # 记录变更文件

# 后续步骤
- 专注区扫描：只处理变更文件
- 收件箱处理：检查是否有变更再处理
```

**实施**：
- 修改启动序列步骤 1
- 更新专注区同步规则（增量模式 + 全量模式）
- 更新收件箱处理（增量检测）

**效果**：
- 无远程更新：节省 60-70%
- 少量变更：节省 50-80%
- 大量变更：自动降级为全量模式

---

### 第三阶段：三种启动模式

#### 5. 分层启动架构
**问题**：所有场景都用同一套启动流程，过于重量级

**方案**：
```
轻量模式（快速启动）：
  - 只加载 CLAUDE.md + _本周.md
  - 跳过 git pull、专注区扫描、记忆系统
  - 适合快速查询、临时讨论

标准模式（启动）：
  - 执行增量同步
  - 缓存 SOUL.md
  - 限量读取 MEMORY_LOG
  - 适合日常工作

深度模式（深度启动）：
  - 全量加载所有文件
  - 强制重新加载 SOUL.md
  - 主动加载 USER.md + MEMORY.md
  - 适合周复盘、重要决策
```

**实施**：
- 新增触发词映射表
- 重构启动序列为三个模式
- 增加模式切换机制

**效果**：
- 轻量模式：节省 90%
- 标准模式：节省 50-60%
- 深度模式：0%（保持完整）

---

### 第四阶段：CLAUDE.md 分层加载

#### 6. 核心 + 扩展架构
**问题**：CLAUDE.md 太长（~435 行），每次都完整加载

**方案**：
```
CLAUDE.md（核心）：
  - ~200 行，约 1500 tokens
  - 包含启动流程、基础规则
  - Claude Code 自动加载

CLAUDE_EXTENDED.md（扩展）：
  - ~300 行，约 2500 tokens
  - 包含详细规则、完整协议
  - 深度模式或按需加载
```

**实施**：
1. 创建 CLAUDE_EXTENDED.md
2. 将详细规则移入扩展文件：
   - 专注区状态同步详细规则
   - 区域代理完整说明
   - 记忆系统详细协议
   - 收件箱处理详细流程
   - 搭档碰头风格指南
   - 技能系统详细规则
3. 精简 CLAUDE.md，保留核心指令
4. 用"详细规则见 CLAUDE_EXTENDED.md"替代

**效果**：
- 日常使用节省 ~2000 tokens（60-70%）
- 深度模式加载完整规则（0% 节省）

## 📊 综合优化效果

### Token 消耗对比

| 场景 | 优化前 | 优化后 | 节省 |
|------|--------|--------|------|
| 快速对话（轻量模式） | 5500 | 1400 | **75%** |
| 日常工作（标准模式） | 5500 | 2500 | **55%** |
| 周复盘（深度模式） | 5500 | 5500 | 0% |

### 实际使用场景

**一周 10 次启动**（1 次周一，2 次深度，7 次标准）：
- 优化前：10 × 5500 = 55,000 tokens
- 优化后：1 × 3500 + 2 × 5500 + 7 × 2500 = 32,000 tokens
- **综合节省：42%**

如果其中 5 次用轻量模式：
- 优化后：1 × 3500 + 2 × 5500 + 2 × 2500 + 5 × 1400 = 25,500 tokens
- **综合节省：54%**

## 💡 关键洞察

### 1. 知识库与项目工作应该分离

**发现**：
- 知识库（memory-work）适合沉淀知识、反思、复盘
- 实际项目工作应该在项目自己的目录中进行
- 混在一起导致上下文混乱、token 浪费

**新架构**：
```
~/Desktop/
├── memory-work/              ← 个人知识库
│   ├── CLAUDE.md             ← 知识管理规则
│   ├── 00 专注区/            ← 周工作台
│   ├── 02 我的知识/          ← 知识沉淀
│   └── 01 项目索引/          ← 项目文档和链接（不含代码）
│
├── project-a/                ← 实际项目 A
│   ├── src/                  ← 源代码
│   └── .claude/              ← 项目专属配置
│
└── project-b/                ← 实际项目 B
```

**原则**：
- 知识库：个人成长、知识沉淀、周期性更新
- 项目目录：交付成果、实际产出、持续性开发
- 跨系统引用：通过文件链接和索引

---

### 2. 分层加载是高性价比优化

**核心思想**：
- 不是所有场景都需要完整上下文
- 80% 的对话只需要 20% 的规则
- 按需加载 > 一次性加载

**实施要点**：
- 核心文件：精简、高频、必需
- 扩展文件：详细、低频、按需
- 自动降级：需要时主动加载

---

### 3. 增量同步是日常最有感的优化

**原因**：
- 大多数时候远程无更新或只有少量变更
- 全量扫描浪费严重
- git diff 提供精确的变更范围

**效果**：
- 无更新场景：节省 60-70%
- 少量变更：节省 50-80%
- 不影响功能完整性（大量变更自动降级）

---

### 4. 缓存机制适用于稳定文件

**适用条件**：
- 文件变更频率低（如 SOUL.md）
- 内容相对稳定
- 有明确的失效检测机制（修改时间）

**不适用**：
- 高频变更文件（如 _本周.md）
- 动态生成内容
- 需要实时同步的数据

## 🚀 未来优化方向

### 高价值优化（推荐实施）

#### 1. _本周.md 智能截取
**问题**：周后期文件越来越长，但很多是已完成任务

**方案**：
- 只读取 YAML frontmatter + 未完成任务
- 已完成任务折叠为计数
- 进展记录只读最近 3 天

**预计节省**：40-60%（周后期）

---

#### 2. 会话状态持久化
**问题**：连续对话每次都重新加载

**方案**：
- 记录上次会话时间
- 距离 < 2 小时提示"继续上次会话"
- 继续模式：只同步变更

**预计节省**：70-80%（连续对话场景）

---

#### 3. 文件大小过滤
**问题**：深度回溯时读取所有文件前 30 行，大文件浪费

**方案**：
- 优先读取 YAML frontmatter
- 超过 500 行的文件只读 frontmatter
- 需要详细内容时再按需加载

**预计节省**：20-30%

---

### 中价值优化（有空再做）

#### 4. 按主题的微启动模式
- 代码模式：只加载项目文件
- 写作模式：只加载写作区 + 知识库
- 学习模式：只加载知识库 + 收件箱
- 规划模式：只加载专注区 + 本周文件

#### 5. MEMORY.md 索引化
- 头部维护索引区块
- 记录关键词、主题、强度
- 启动时只读索引，需要时再加载具体条目

---

### 长期优化（需要重构）

#### 6. 智能上下文窗口管理
- 动态评估哪些上下文还需要

## 🎓 经验总结

### 优化方法论

1. **先测量，再优化**
   - 识别 token 消耗最多的地方
   - 量化优化效果
   - 避免过早优化

2. **分层思维**
   - 核心 vs 扩展
   - 必需 vs 可选
   - 高频 vs 低频

3. **增量优先**
   - 只处理变更部分
   - 避免重复加载
   - 保留全量模式作为兜底

4. **按需加载**
   - 不是所有场景都需要完整上下文
   - 自动降级机制
   - 用户可主动切换模式

5. **保持功能完整性**
   - 优化不能牺牲功能
   - 深度场景保留完整能力
   - 提供多种模式选择

---

### 架构设计原则

1. **关注点分离**
   - 知识管理 ≠ 项目开发
   - 个人成长 ≠ 交付成果
   - 不同场景用不同工具

2. **单一职责**
   - 每个文件有明确定位
   - 核心文件保持精简
   - 详细规则独立存储

3. **可扩展性**
   - 新增模式不影响现有模式
   - 扩展文件可以继续拆分
   - 保留未来优化空间

4. **用户体验优先**
   - 默认行为符合 80% 场景
   - 提供简单的切换机制
   - 自动化 > 手动配置
