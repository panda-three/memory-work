# 技能库区域代理

## 区域定位

**技能（Skill）** 是可执行的能力模块，比静态工具更强。一个技能可以：
- 运行脚本或命令
- 生成输出（日历、报告、分析）
- 触发工作流或流程
- 管理系统操作
- 对接外部服务

技能是知识系统与实际自动化之间的桥梁。

## 目录结构

```
06 我的工具/
├── datetime-check/
│   ├── SKILL.md              ← 技能说明书（必须）
│   ├── scripts/              ← 可执行脚本（可选）
│   ├── references/           ← 参考资料（可选）
│   └── assets/               ← 模板素材（可选）
├── memory-review/
│   ├── SKILL.md
│   └── ...
├── read-url/
│   ├── SKILL.md
│   └── ...
└── 00.skills_agent.md        ← 本文件（区域代理）
```

### 单个技能的文件结构

```
skill-name/
├── SKILL.md               ← 必须，技能说明书
│   ├── YAML frontmatter
│   ├── 用途
│   ├── 触发方式
│   ├── 输入参数
│   ├── 输出格式
│   ├── 依赖项
│   └── 执行指令
├── scripts/               ← 可选，可执行脚本
├── references/            ← 可选，参考资料
└── assets/                ← 可选，模板素材
```

## SKILL.md 格式

### Frontmatter（必填）

```yaml
---
name: skill-name           # kebab-case 命名
type: protocol | skill     # 协议型 或 技能型
trigger: 触发条件描述
version: 1.0
created: YYYY-MM-DD
---
```

### 正文结构

```markdown
# 技能名称

## 用途
这个技能做什么、为什么需要它。

## 触发方式
什么时候运行、谁来触发。

## 输入参数
| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| ... | ... | ... | ... |

## 输出格式
返回什么、格式是什么。

## 依赖项
需要什么环境、工具或其他技能。

## 执行指令
AI 读到这里后，按以下步骤执行：
1. ...
2. ...
```

## 两种运行机制

### 协议型（Protocol）—— 条件反射

- 在 CLAUDE.md 启动序列中定义
- **每次会话自动执行**，无需用户触发
- 例：datetime-check、专注区同步
- 类比：后台反射，必须执行，不可跳过

### 技能型（Skill）—— 有意识的动作

- AI 根据用户意图语义匹配触发
- 例："保存对话"、"生成日历"、"做个复盘"
- 用户不需要说出精确的触发词，AI 理解意图即可
- 类比：按需调用，灵活适应

## AI 在此区域的行为规则

- **创建**：写 SKILL.md 之前先向用户提议（说明用途和触发方式）
- **修改**：更新已有技能时做版本管理
- **测试**：新技能注册前先验证可用
- **去重**：创建前检查是否已有类似技能

## 最佳实践

1. **单一职责**：一个技能 = 一个能力
2. **触发清晰**：明确定义何时、如何运行
3. **参数可复用**：接受输入，不硬编码
4. **优雅容错**：失败时给出有用的错误信息
5. **有文档**：每个技能都要有清晰的用途和用法说明
6. **有版本**：追踪变更，保持兼容
7. **有示例**：SKILL.md 中包含使用示例

## Token 优化：Skill 的成本意识（重要）

### 第一性原理

Token 消耗 = 上下文窗口中的所有内容 × 对话轮次。具体到 Skill：

```
单次 Skill 调用的 Token 成本 = SKILL.md + 读取的 references/ + 读取的 assets/ + 多轮对话累积
```

**关键认知**：references/ 和 assets/ 是保证输出质量的核心资产，不能为了省 Token 而删除。优化方向不是"删内容"，而是"控制加载方式和对话长度"。

### 错误做法（踩过的坑）

- ❌ 把 references/ 内容压缩成几句话塞进 SKILL.md → 丢失细节，输出质量下降
- ❌ 删除 assets/ 模板让 AI 自由发挥 → 输出格式不稳定
- ❌ 为了省 Token 把所有内容内联到一个文件 → SKILL.md 膨胀，且每次调用都全量加载

### 正确的优化策略

**1. 分阶段按需加载（最重要）**

SKILL.md 的执行指令中，references/ 应该在对应阶段才读取，而不是一开始全部读入：

```
阶段 1：容量评估 → 只读 references/00-容量评估.md
阶段 2：课程设计 → 只读 references/01-内容规划.md + 02-时间分配.md + 03-练习设计.md
阶段 3：生成辅助材料 → 只读 assets/ 中需要的模板
```

如果用户只需要容量评估，就不会加载后续阶段的文件。

**2. 控制对话轮次**

Skill 执行过程中，减少不必要的确认轮次。每多一轮对话，之前读取的所有内容都要重新发送。

- 能一次输出的不要分多次
- 阶段之间的确认点控制在 1-2 个

**3. 新建 Skill 时的成本预算**

| Skill 规模 | SKILL.md | references/ | 预估单次成本 | 适用场景 |
|------------|----------|-------------|-------------|---------|
| 轻量 | ≤ 3K | 无 | ~750 tokens | 简单工具（datetime-check） |
| 标准 | ≤ 6K | ≤ 15K | ~5,000 tokens | 常规任务（read-url） |
| 重型 | ≤ 6K | ≤ 30K | ~10,000 tokens | 复杂流程（lesson-planner） |

重型 Skill 不是问题，但要意识到成本，并通过分阶段加载来控制。

**4. 使用 skill-creator 创建 Skill**

自由发挥容易写出结构不规范、内容冗余的 Skill。优先使用 skill-creator 技能来创建新 Skill，它会引导标准化的创建流程，避免不必要的膨胀。

### references/ 和 assets/ 的定位

- **references/**：存放保证输出质量的参考标准和细节规则。这些是 AI 通用知识无法替代的"你的特定要求"
- **assets/**：存放需要精确格式的输出模板。保证输出一致性
- **scripts/**：可执行脚本。不进入上下文，不消耗 Token，放心使用

## 组织规则

- 按字母或功能分组
- 命名统一用 `kebab-case`（如 `datetime-check`）
- 文件结构保持一致
- `scripts/` 与文档分开存放
- 旧版本技能加时间戳归档
